# Storage Strategy

## Hybrid Storage Model

We use a **hybrid approach**: frequently accessed data locally, archive data in cloud.

```
OCI Block Volume (200GB)          Backblaze B2
├─ Thumbnails  ←───────────────  ├─ Original Photos
├─ Database                    │  ├─ Documents
├─ Cache                       │  ├─ Backups
└─ Ephemeral                   │  └─ Long-term archive
     │                                │
     └── Fast, local              └── Cheap, durable
```

## Why This Approach?

**OCI Block Volume**:
- ✅ Fast SSD I/O
- ✅ Free (Always Free tier)
- ❌ Limited to 200GB
- ❌ $0.10/GB/month if exceeded

**Backblaze B2**:
- ✅ $0.006/GB/month (~$6/TB)
- ✅ 99.999999999% durability
- ✅ No egress fees to OCI
- ❌ Slightly slower access

## Storage Layout

```
/data/
├── immich/
│   ├── thumbnails/     # Generated by Immich
│   ├── cache/          # Temporary files
│   └── b2-mount/       # Mounted B2 bucket (read-only)
│       └── upload/     # Original photos
└── backups/
    ├── postgres/       # Database dumps
    ├── configs/        # Configuration backups
    └── weekly/         # Full snapshots
```

## Optimization Strategies

### Minimize B2 API Calls

**Thumbnails locally**:
- Serve hot data from OCI SSD
- Reduces B2 Class B transactions
- Fast photo browsing experience

**Batch uploads**:
- Immich uploads originals directly to B2
- Configurable upload batch sizes

### Cache Strategy

**rclone vfs-cache**:
```
--vfs-cache-mode writes
--vfs-cache-max-size 10G
--dir-cache-time 72h
```

This keeps recently accessed files local.

## Disaster Recovery

**Scenario**: OCI instance fails

1. **Data safety**: All photos in B2 (safe)
2. **Restore time**: 30-60 minutes
3. **Process**:
   - Provision new OCI instance
   - Mount B2 bucket
   - Restore database from B2 backup
   - Regenerate thumbnails (automatic)

**Worst case**: Lost data = changes since last backup (max 24 hours)
