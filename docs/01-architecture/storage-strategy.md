# Storage Strategy

## B2-Primary Storage Model

All original photos and irreplaceable data are stored in **Backblaze B2**. OCI local storage is used only for caches, thumbnails, and databases — data that can be regenerated or restored from B2.

This ensures we stay within OCI's free tier while having virtually unlimited photo storage.

```
OCI Block Volume (200GB)          Backblaze B2
├─ Thumbnails  (regenerable)    ├─ Original Photos (primary)
├─ Database    (backed up)      ├─ Database Backups
├─ Cache       (ephemeral)      ├─ Config Backups
└─ ML models   (re-downloadable)└─ Terraform State
     │                                │
     └── Fast, local, free        └── Cheap, durable ($6/TB)
```

## Why This Approach?

**OCI Block Volume**:
- Fast SSD I/O
- Free (Always Free tier)
- Limited to 200GB
- $0.10/GB/month if exceeded — we avoid this

**Backblaze B2**:
- $0.006/GB/month (~$6/TB)
- 99.999999999% durability
- No egress fees to OCI
- Slightly slower access (mitigated by rclone VFS cache)

## How Photos Get to B2

Immich writes uploads to a directory that is an **rclone FUSE mount** of your B2 bucket. This means:

1. User uploads photo via Immich
2. Immich writes to `/data/immich/b2-mount/upload/`
3. rclone transparently writes to B2
4. VFS write-back cache handles the latency

The rclone mount is configured as a systemd service with:
```
--vfs-cache-mode writes    # Cache writes locally, then upload
--vfs-cache-max-size 10G   # Up to 10GB local write cache
--buffer-size 256M         # 256MB read buffer
--dir-cache-time 72h       # Cache directory listings for 3 days
```

Thumbnails are generated by Immich and stored locally on the block volume at `/data/immich/thumbnails/` for fast browsing.

## Storage Layout

```
/data/                             # Block volume mount
├── immich/
│   ├── thumbnails/                # Generated by Immich (local only)
│   ├── cache/                     # Temporary files
│   └── b2-mount/                  # rclone FUSE mount of B2 bucket
│       └── upload/                # Original photos (stored in B2)
└── backups/
    ├── postgres/                  # Database dumps (synced to B2)
    ├── configs/                   # Configuration backups
    └── weekly/                    # Full snapshots

B2 Bucket (your-bucket-name)/
├── photos/                        # rclone mount target
│   └── upload/                    # Immich uploads land here
├── backups/
│   ├── postgres/                  # Daily DB dumps
│   ├── configs/                   # Hourly config snapshots
│   ├── weekly/                    # Weekly full backups
│   └── certs/                     # SSL certificates
└── terraform/                     # Terraform state backups
```

## Disaster Recovery

**Scenario**: OCI instance fails

1. **Data safety**: All photos in B2 (safe)
2. **Restore time**: 30-60 minutes
3. **Process**:
   - Run `tofu apply` to provision new instance (cloud-init handles base setup)
   - SSH in, configure `.env` files
   - Run `install.sh` to set up B2 mount and cron jobs
   - Restore database from B2 backup
   - Start services — thumbnails regenerate automatically from B2 originals

**Worst case**: Lost data = changes since last backup (max 24 hours for DB, photos are real-time via B2 mount)
